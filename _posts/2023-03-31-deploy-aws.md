# Deploy 배포
---
<br>

![AWS image](/public/img/AWS/aws.jpeg){: width="800px"}

<br>
지금까지는 로컬 환경에서 코드를 작성했다. 내가 작성한 코드가 그저 로컬 환경에서 구동된다면, <br>
다른 사람들은 내가 개발한 서비스를 이용할 수 없을 것이다. <br>
사용자가 서비스를 이용할 수 있게 하는 ‘배포’에 대해서 알아보자.<br>
<br>
배포란 내가 개발한 서비스를 사용자들이 이용 가능하게 하는 일련의 과정이다. <br>
회사마다 추가적인 과정이 있을 수 있지만, 기본적으로 4단계를 거쳐서 개발한 서비스를 배포하게 된다.<br>
<br>

`Development` 단계는 각자의 컴퓨터에서 코드를 작성하고 테스트하는 과정이다.<br>
개발 단계이기 때문에 실제 데이터를 이용하지 않고 더미 데이터를 이용해서 테스트한다.<br>
<br>

`Integration` 단계는 각자의 컴퓨터에서 작성한 코드를 합치는 과정이다.<br>
내가 작성한 코드가 다른 코드를 침범해서 오류를 일으키지 않는지, 코드 간에 conflict가 있지는 않은지 확인하는 과정을 거친다.<br>
<br>
`Staging` 단계에서는 실제 출시 단계인 `Production` 단계와 가장 유사한 환경에서 테스트를 진행한다.<br>
실제 데이터를 복사해서 문제가 없는지 등에 대해 다양한 환경에서 테스트를 진행한다.<br>
또한 서비스와 관련된 부서 혹은 인원의 확인 과정을 거친다. 예를 들면 작성된 코드가 마케팅팀 혹은 디자인팀이 예상했던 결과와 일치하는지 확인하는 과정이다.<br>
<br>
`Production` 단계는 개발된 서비스를 출시하는 단계다.<br>
사용자가 접속할 수 있는 Production 환경에서 코드를 구동하고 서비스를 제공한다.<br>
실제 데이터를 가지고 서비스가 운영되기 때문에 문제가 생기면 안 되는 단계다.<br>
<br>
Development 환경과 Production 환경은 서로 다를 수 있다.<br>
<br>
내가 개발부터 배포까지 모든 것을 통제할 수 있는 상황이라면, 크게 걱정 없이 Production 환경을 구성할 수 있을 것이다.<br>
그러나, 여러 명이 함께 작업하는 프로젝트라면 어떨까? node 버전도 제각각일 것이고, 인증 정보나 데이터베이스 등에 접근하기 위해 사용하는 엔드포인트도 제각각일 것이다.<br>
<br>
예를 들어보자. 내 로컬에 설치된 데이터베이스 비밀번호는 헬로1234! 인 데, 클라우드에 설치된 데이터베이스 비밀번호는 secret! 일 수 있다.<br>
이처럼 Development 환경과 Production 환경은 서로 다를 수 있다.<br>
<br>
따라서 배포 단계에서는, 환경의 차이를 이해하고 환경 설정과 코드를 분리하는 것이 중요하다.<br>
<br>
작성한 코드가 다른 환경에서 정상 작동할 수 있게 하려면, 설정을 환경 변수(envvars나 env라고도 불림)에 저장해야 한다. <br>
환경 변수는 코드 변경 없이도 배포 때마다 쉽게 변경할 수 있다. 설정 파일과 달리, 잘못해서 코드 저장소에 올라갈 가능성도 낮다.<br>
<br>
애플리케이션의 모든 설정이 정상적으로 코드 바깥으로 분리되어 있는지 확인할 수 있는 간단한 방법은, <br>
어떠한 인증정보도 유출시키지 않고 코드가 지금 당장 오픈 소스가 될 수 있는지 확인하는 것이다.<br>
<br>
코드 상의 모든 곳에 절대 경로가 아닌 상대 경로를 사용해야 하며, `.env` 등을 이용해 환경 변수를 설정해야 한다. <br>
그 외에도 docker와 같은 가상화 도구는 환경 자체를 메타데이터에 담아서 모든 개발 환경을 통일시킨다.<br>
<br>
배포를 위해 존재하는 다양한 플랫폼이 있다. 그 중에서도 AWS의 여러 서비스와 연결지어가며 배포에 대해 공부해보자.<br>
<br>
<br>

# Deploy Strategy 배포 전략
---

지금까지 나는 로컬 환경에서 클라이언트와 서버, 데이터베이스를 띄워서 작성한 코드를 구동하고 확인했다.<br>
<br>
내가 개발한 서비스에 외부 사용자들이 접속할 수 있게 하려면 어떻게 해야 할까?<br>
<br>
개발한 서비스를 사용자가 이용할 수 있도록 하는 것을 배포라고 한다.<br>
어떻게 사용자들이 내 서비스를 이용할 수 있게 할 수 있을까?<br>
사용자들에게 localhost:3000으로 들어오라고 할 수는 없는 노릇이다.<br>
<br>
그렇다면 사용자들에게 Client를 어떻게 제공할지,<br>
그리고 Client를 받은 사용자들에게 서비스를 이용하기 위한 Server를 어떻게 제공할 것인지,<br>
Server의 데이터를 저장하고 제공할 Database는 어떻게 제공할 것인지 생각해 봐야 한다.<br>
<br>
작성한 Client 코드를 사용자들에게 어떻게 제공할지 생각해 보자.<br>
<br>
AWS에서 제공하는 서비스인 S3라는 서비스를 통해 사용자들에게 Client를 제공할 수 있다.<br>
<br>
로컬 환경에서는 자체 개발 서버(ex. create-react-app)를 이용해서 클라이언트 앱을 실행시키는 것이 보통이다. <br>
그럼, 클라이언트를 위해서 EC2 인스턴스를 사용해야 할까?<br>
<br>
그렇지 않다. 클라이언트 앱을 정적 파일로 빌드하여 제공한다. <br>
따라서 S3를 이용해서 클라이언트를 배포한다.<br>
<br>
이때 필요한 것이 빌드다.<br>
<br>
빌드란 쉽게 말해서 불필요한 데이터를 없애고, 여러 갈래로 퍼져있는 데이터들을 통합/ 압축하여 배포하기에 최적화된 상태를 만드는 것이다. <br>
빌드 과정을 진행하기 전과 비교했을 때 데이터의 용량이 줄어들고, 웹 사이트의 로딩 속도가 빨라진다는 장점이 있다. <br>
<br>
하지만 일반적인 의미에서 빌드는 소스코드를 실행 가능한 번들로 변환하는 컴파일 과정을 의미한다. <br>
웹 앱에서와같이 HTML, CSS, JS의 형태로 배포하는 경우는 조금 다르다. 웹 앱은 배포 가능한 정적 파일(static files)의 형태로 만들어 줘야 한다.<br>
<br>
asset 자체가 정적인 경우, 있는 그대로 배포하면 된다. <br>
React의 경우 npm run build와 같은 명령을 사용해서, 정적 파일 형태의 결과물을 만들어 낸 후 배포하면 된다. <br>
사용하고 있는 환경에 따라 빌드 과정은 다를 수 있다.<br>
<br>
S3로 사용자들에게 Client Application을 제공하고 있는데, 사용자가 지구 반대편에 있다면 어떻게 빠르게 서비스를 제공할 수 있을까?<br>
<br>
AWS에서 제공하는 CDN 서비스인 CloudFront를 통해서 각지의 데이터 센터에 데이터를 분산시켜서 저장해 뒀다가 가까운 지역에서 데이터를 주는 방식으로, <br>
사용자에게 보다 빠른 서비스를 제공할 수 있다.<br>
<br>
Client Application을 통해서 요청을 전달할 Server Application은 어떻게 배포해야 할까?<br>
<br>
안정적으로 서비스를 제공하기 위해 가상의 PC(AWS EC2)를 빌려 서버 코드를 구동할 수 있다. AWS는 Database 특화 서비스인 RDS 서비스를 제공하고 있다.<br>
AWS가 유지 보수하는 RDS를 이용해 즉시 데이터베이스를 사용할 수 있다.<br>
RDS 서비스를 이용하면 EC2를 통해 배포된 Server Application의 데이터를 저장, 제공하는 데이터베이스를 배포할 수 있다.<br>
<br>
우리는 지금까지 `www.google.com`과 같은 도메인 주소를 이용해서 서비스에 접근했다.<br>
google 사이트에 접속하기 위해 `172.217.161.228`이라는 IP 주소를 입력하지는 않았다.<br>
지금 막 배포된 서비스에 도메인 주소를 통해 접근할 수 있을까?<br>
S3, EC2를 이용해서 배포된 서비스는 내 서비스와는 전혀 상관없는 긴 도메인 주소(혹은 IP 주소)를 통해 이용해야 한다.<br>
<br>
TodoList 서비스를 제공한다고 생각해 보자.<br>
<br>

`www.todolist.ap-northeast-2.compute.amazonaws.com` 주소보다는 `www.todolist.com` 주소 일 때,<br>
직관적으로 서비스를 이해할 수 있고 쉽게 접근할 수 있다.<br>
<br>
AWS에서 제공하는 Route 53 서비스를 이용하면 직관적인 도메인 주소를 통해 서비스에 접근하도록 할 수 있다.<br>

# AWS
---

## EC2
<br>
EC2는 컴퓨터를 한 대 빌리는 것이므로 컴퓨터가 하는 모든 일을 할 수 있다.<br>
<br>
빌린 컴퓨터는 직접 사용하는 컴퓨터와 다르게, 아마존 데이터 센터(인프라)에 만들어져 있기 때문에<br>
컴퓨터를 조작하기 위해 네트워크(인터넷)를 통해서 컴퓨터를 제어해야 한다는 차이점이 있을 뿐 일반적인 컴퓨터와 다른 점은 없다.<br>
<br>
아마존 EC2를 통해서 할 수 있는 가장 기본적인 일은<br>
웹서버를 설치하고 웹 서버를 통해서 사용자가 웹 브라우저를 통해 요청하는 서비스를 제공하는 것이다.<br>
인스턴스는 1대의 컴퓨터를 의미하는 단위이고 AWS에서 컴퓨터를 빌리는 것을 ‘인스턴스를 생성한다’ 라고 한다.<br>
<br>
AMI는 소프트웨어 구성이 기재된 템플릿이다.<br>
이미지 종류로는 단순히 운영체제(윈도우, 우분투 리눅스 등)만 깔려있는 템플릿을 선택할 수도 있고, <br>
아예 특정 런타임이 설치되어 있는 템플릿이 제공되는 경우도 있다. (우분투 + node.js, 윈도우 + JVM 등)<br>
<br>
AWS에서 빌릴 PC는 그 용도에 맞게 운영체제, 런타임 등이 구성된 Setting을 선택할 수 있다.<br>
인스턴스는 선택한 AMI를 토대로 구성된다.<br>
AWS에는 상당히 많은 양의 AMI 세팅이 준비되어 있기 때문에 손쉽게 인스턴스의 운영체제를 구성할 수 있다.<br>
<br>
세팅되어 있는 AMI 이외에도 필요에 따라 직접 AMI를 구성할 수도 있다.<br>
AWS EC2 인스턴스를 생성한다는 것은 AMI를 토대로 운영체제, CPU, RAM 혹은 런타임 등이 구성된 컴퓨터를 빌리는 것이다.<br>
<br>
<br>

## RDS
<br>
RDS는 Relational Database Service의 약자로, AWS에서 제공하는 관계형 데이터베이스 서비스다. <br>
<br>
왜 RDS를 사용할까? EC2는 가상의 컴퓨터를 임대하는 서비스다. <br>
EC2 인스턴스에 MySQL 같은 관계형 데이터베이스 엔진을 설치하면 굳이 RDS를 사용할 이유가 없지 않을까? <br>
데이터베이스만 따로 분리해서 서비스를 이용해야 할 이유가 있을까?<br>
<br>
EC2 인스턴스에 관계형 데이터베이스 엔진을 설치해서 데이터를 관리할 때와,<br> 
RDS를 통해 데이터를 관리할 때의 차이는 개인 소유 차량과 렌터카 회사에서 대여한 차량으로 비유할 수 있다.<br>
<br>
EC2 인스턴스에 데이터베이스를 설치하여 데이터를 관리하는 것은 개인 소유 차량을 이용하는 것과 비슷하다.<br>
 개인 소유 차량을 이용하면 유지 보수, 보험처리 같은 일들을 온전히 운전자가 부담한다. <br>
 차량 정비를 위해서 정비소에 주기적으로 방문해야 하고, 기타 차량과 관련된 다른 일이 생길 때 들여야 하는 시간과 수고가 크다.<br>
<br>
EC2 인스턴스를 사용하면 데이터베이스와 관련해서 자동으로 관리를 담당하는 부분이 매우 적기 때문에, <br>
사용자가 일일이 시간을 투자하여 데이터베이스 엔진의 설치와 버전 관리, 데이터 백업을 해야 한다.<br>
 <br>
게다가 가용성과 내구성이 확보되지 않아 데이터베이스에 저장된 데이터가 유실되거나 정상적으로 사용할 수 없게 될 확률이 커지며, <br>
후에 필요에 따라 데이터베이스의 규모를 확장하기 어렵다.<br>
<br>
그에 비해 RDS를 이용하는 것은 렌터카 회사에서 차량을 대여하는 것과 비슷하다.<br>
렌터카 회사에서 차량을 대여하면 대여 차량과 관련하여 시간이 들어가는 일들을 렌터카 회사에서 대신 처리한다. <br>
운전자는 차량을 관리하는 일에 대해서 시간을 따로 쏟을 필요 없이 운전만 하면 되기에 매우 편리하다.<br>
<br>
RDS를 이용하면 데이터베이스 유지 보수와 관련된 일들을 RDS에서 전적으로 자동 관리한다. <br>
사용자가 해야 할 일은 초기 설정을 제외하고 데이터베이스에 저장된 데이터를 관리하는 일 밖에 없기에 큰 편의성을 느낄 수 있다.<br>
<br>
기타 RDS 이용 시 얻을 수 있는 장점으로는 다양한 데이터베이스 엔진 선택지를 제공한다는 점이 있다. <br>
회사에서 근무하고 있는 실무자는 회사에 필요한 데이터베이스 엔진을 취사선택하여 이용할 수 있다. <br>
그 외 일반 사용자는 데이터베이스 엔진마다 제공하는 기능이 조금씩 다르기에 필요와 목적에 맞게 데이터베이스 엔진을 선택하여 효율성을 높일 수 있다.<br>
<br>

## S3
<br>
클라우드 스토리지란 인터넷 공간에 데이터를 저장하는 저장소다. 컴퓨터 부품으로 비유하면 하드디스크의 역할을 하는 서비스다. <br>
클라우드 스토리지 서비스는 뛰어난 접근성을 가지고 있다. 컴퓨터의 하드디스크에 저장된 파일에 접근하기 위해서는 해당 컴퓨터를 이용해야만 한다. <br>
그러나 클라우드 스토리지를 이용하면 웹 환경이라면 언제 어디서나 저장된 파일에 접근할 수 있다.<br>
또한 컴퓨터 뿐만 아니라 웹에 접속이 가능한 다른 전자기기를 활용하여 클라우드 스토리지에 저장된 데이터에 접속할 수 있다.<br>
<br>
S3는 Simple Storage Service의 약자로 AWS에서 제공하는 클라우드 스토리지 서비스다.<br>
뛰어난 접근성 외에도 S3 사용 시 얻을 수 있는 이점은 여러가지다.<br>
<br>
<br>

### 높은 확장성
<br>
먼저 S3 사용 시 얻을 수 있는 이점으로 높은 확장성이 있다. 확장성이 높으면 많은 시간과 수고를 들이지 않고 스토리지 규모를 확장/축소할 수 있다.<br>
또한 S3에서는 스토리지의 용량을 무한히 확장할 수 있다. 그리고 사용한 만큼만 비용을 지불하면 되기 때문에 비용 측면에서 매우 효율적이다.<br>

<br>

### 강력한 내구성
<br>
스토리지의 내구성이 높으면 저장된 파일을 유실할 가능성이 적어진다. S3는 99.999999999%의 내구성을 보장한다. <br>
S3 스토리지의 내구성에 대한 실생활의 예를 하나 들자면, S3에 저장된 파일을 잃어버릴 확률보다, <br>
길을 걷다가 벼락을 맞을 확률(약 0.0000007%의 확률)이 700배나 더 높다. <br>
<br>

### 가용성 보장
<br>
가용성이 높으면 스토리지에 저장된 파일들을 정상적으로 사용할 수 있는 시간이 길어진다. S3는 연간 99.99%의 스토리지 가용성을 보장하도록 설계되어 있다. <br>
이는 1년 동안 S3에 파일을 저장했을 시, 스토리지를 이용하는 데 장애가 발생하는 시간이 8.76시간 밖에 되지 않는다는 뜻이다.<br>
AWS는 어떤 원리로 해당 서비스들의 높은 가용성과 내구성을 보장할 수 있는 걸까?<br>
<br>

![AWS region](/public/img/AWS/region.jpeg){: width="800px"}

<br>
위 지도를 보면 주황색 동그라미가 쳐진 지역이 있다. 이 지역을 '리전(Region)'이라고 부른다. <br>
리전이란 AWS에서 클라우드 서비스를 제공하기 위해서 운영하는 물리적인 서버의 위치를 뜻한다.<br>
그리고 주황색 동그라미 안에는 숫자가 있는데, 이 숫자는 리전에 위치한 가용 영역의 수를 뜻한다. <br>
가용 영역(Availability Zone)이란 각 리전 안에 존재하는 데이터 센터(IDC)를 뜻한다. 가용 영역은 개별적인 위치에 떨어져 존재한다. <br>
그래서 한 곳의 가용 영역이 재난이나 사고로 인해 가동이 불가능해지더라도 다른 가용 영역에 백업을 해놓은 데이터를 활용하여 문제없이 서버가 가동되게 한다. <br>
이런 가동 방식 덕분에 AWS에서 제공하는 서비스들은 높은 가용성과 내구성을 보장한다.<br>
<br>

### 다양한 스토리지 클래스 제공
<br>
S3는 다양한 스토리지 클래스를 제공한다. 저장소를 어떤 목적으로 활용할지에 따라 선택할 수 있는 스토리지 클래스가 달라진다. <br>
S3 사용자들이 대표적으로 많이 선택하는 스토리지 클래스는 두 가지가 있다. Standard 클래스와 Glacier 클래스이다. <br>
<br>
Standard 클래스는 범용적인 목적으로 사용하기 좋다. 데이터에 빠른 속도로 접근할 수 있고, 데이터 액세스 요청에 대한 처리 속도가 빠르다.<br>
대신 데이터를 오래 보관하는 목적으로는 효율적인 선택지가 아니다. 보관 비용이 높게 발생하기 때문이다.<br>
<br>
장기적인 보관 목적으로 스토리지를 사용할 때는 Glacier를 사용하는 것이 효율적이다. <br>
비록 저장된 데이터에 액세스하는 속도는 느리지만, 데이터를 보관하는 비용이 매우 저렴하다는 장점이 있다. <br>
이 외에도 Standard-IA, One Zone-IA, S3 Glacier Deep Archive 등등 여러 가지 스토리지 클래스가 존재하여 <br>
사용자의 이용 목적에 따라 다양한 스토리지 클래스를 사용할 수 있다.<br>
<br>

### 정적 웹 사이트 호스팅
<br>
먼저 정적 웹 사이트 호스팅이 무엇인지 알기 위해 '정적' 파일에 대한 이해가 선행되어야 한다. <br>
정적 파일은 서버의 개입 없이 생성된 파일을 뜻한다. 반대로 클라이언트가 서버에 요청을 보내면, <br>
서버가 요청에 맞추어 그 자리에서 생성한 파일을 '동적' 파일이라고 부른다.<br>
<br>
웹 호스팅이란 서버의 한 공간을 임대해 주는 서비스를 뜻한다. 구글에 '웹 호스팅'이란 단어를 검색하면 여러 웹 호스팅 업체의 목록이 뜨는 것을 확인할 수 있다.<br> 
웹 호스팅 업체들을 통해 개인 또는 단체가 웹 호스팅 업체가 제공하는 서버의 한 공간을 빌려서 원하는 서비스를 배포할 수 있다.<br>
<br>
S3에서는 사용자들에게 정적 웹 사이트를 배포할 수 있는 공간을 제공한다. <br>
버킷이라는 저장 공간에 정적 파일을 업로드하고 버킷을 정적 웹 사이트 호스팅 용도로 구성하면 정적 웹 사이트를 배포할 수 있다.<br>
<br>
버킷이란 S3에 저장되는 파일들이 담기는 바구니다. 파일을 저장하는 최상위 디렉토리라고 설명할 수 있다. <br>
S3에서 저장되는 모든 파일은 버킷 안에 저장되어야 하고, 버킷에는 무한한 양의 파일을 저장할 수 있다. <br>
그리고 각각의 버킷은 이름을 가지고 있는데, 버킷의 이름은 버킷이 속해 있는 리전(버킷이 생성된 지역)에서 유일해야 한다. <br>
<br>
또한 버킷 정책을 생성하여 해당 버킷에 대한 다른 유저의 접근 권한을 수정할 수 있다.<br>

S3에서 버킷에 담기는 파일을 객체라고 부른다. 왜 객체라고 부를까? S3에서 저장소에 데이터를 저장할 때 `키-값` 페어 형식으로 데이터를 저장하기 때문이다. <br>
S3에 저장되는 객체는 파일과 메타데이터로 구성된다. 파일은 위에 설명한 대로 `키-값` 페어 형식으로 데이터를 저장한다. <br>
파일의 `값`에는 실제 데이터를 저장한다. S3 객체의 값으로 저장될 수 있는 데이터의 최대 크기는 5TB다. <br>
파일의 `키`는 각각의 객체를 고유하게 만들어주는 식별자 역할을 한다. 파일의 `키`를 이용하여 원하는 객체를 검색할 수 있다.<br>
메타데이터는 객체의 생성일, 크기, 유형과 같은 객체에 대한 정보가 담긴 데이터다. 객체를 설명하는 데이터라고 이해하면 된다.<br>
<br>
모든 객체는 고유한 URL 주소를 가지고 있다.<br>

URL 주소는 `http://[버킷의 이름].S3.amazonaws.com/[객체의 키]`의 형태를 지니고, URL 주소를 통해서도 원하는 데이터에 접근할 수 있다.<br>
